What I Learned from Today's RAG Playbooks

Overview

Today's session helped me clearly understand how a RAG (Retrieval Augmented Generation) system actually works internally instead of just using it like a black box. I gained deep insights into both the architecture and the design decisions that make or break a RAG system.

From the Normal Playbook

The Two Phases of RAG

The most important concept I learned is that RAG works in two different phases, and they should not be confused. Understanding this separation is crucial for building efficient systems.

1

Offline Indexing Phase

In this stage, documents are loaded, split into chunks, converted into embeddings, and stored in a vector database. This process runs only once or whenever new data is added. It does not run every time a question is asked.

2

Online Retrieval Phase

This runs every time a user asks a question. The system converts the question into an embedding, searches for similar chunks in the database, attaches that context to a prompt, and sends it to the LLM to generate the answer. This part needs to be fast and efficient.

📦

Chunking Strategy

If chunk size is too small, important context gets broken. If too large, unrelated information gets mixed in. Overlap helps preserve meaning between chunks, but overlap must always be smaller than chunk size.

🌡️

Temperature = 0

We use temperature = 0 in RAG systems because it reduces randomness and makes the model more factual. RAG systems should focus on correctness, not creativity.

From the Advanced Playbook

Thinking Like a System Designer

The advanced part helped me think more like a system designer, considering scalability, model selection, and production-readiness of RAG pipelines.

Embedding Model Selection

Different models perform better for different domains:

  • General models are good for normal English documents
  • Domain-specific models are needed for medical or legal documents
  • Critical rule: If we change the embedding model, we must re-embed all documents and recreate the vector database
  • Warning: Mixing different embedding dimensions will break retrieval

Advanced Concepts (Continued)

📊

Similarity Metrics

A high cosine similarity score does not always mean the chunk answers the question correctly. Sometimes it just shares similar words but does not contain the required information.

🗄️

Vector Database Selection

Small projects can use simple local databases, but large-scale production systems need managed cloud databases for scaling and reliability.

Database Choice Factors

The choice of vector database depends on multiple factors:

  • Size of data: How many documents and chunks need to be stored
  • Number of users: Concurrent query load and response time requirements
  • Team capability: Self-hosted vs managed services based on expertise

Overall Understanding

Key Takeaway

From today's playbooks, I understood that building a RAG system is not just about connecting documents to an LLM. Every step — chunking, embeddings, similarity search, and database choice — directly affects performance and answer quality.

Most RAG failures happen because of poor design decisions in early stages, not because the LLM is bad. A well-designed pipeline is more important than just writing good prompts.
